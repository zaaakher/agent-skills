# ZATCA Phase 2 TypeScript Integration Skill

## Purpose

This skill teaches an AI agent how to build production-grade ZATCA Phase 2 integrations using:

- TypeScript
- Node.js
- React (optional frontend/admin dashboard)
- PostgreSQL
- Queue-based async architecture

The agent should generate:
- clean architecture
- reusable modules
- production-safe code
- sandbox + production support
- scalable invoice processing
- secure certificate handling

---

# Core Philosophy

The AI should NEVER generate:
- monolithic invoice code
- synchronous checkout-only flows
- insecure certificate storage
- business logic mixed with XML generation
- XML string concatenation hacks

The AI SHOULD generate:
- modular architecture
- strong typing
- async retry pipelines
- deterministic signing
- transaction-safe sequencing
- queue-based submission

# TypeScript Rules

The AI should ALWAYS:
- use strict typing
- avoid any
- generate interfaces/types
- separate DTOs from entities
- validate all external inputs

tsconfig recommendations:
- strict: true
- noImplicitAny: true
- exactOptionalPropertyTypes: true

---

# Recommended Invoice Domain Model

The AI should generate domain types like:

```ts
interface Invoice {
  uuid: string;
  invoiceNumber: string;
  issueDate: string;
  seller: Seller;
  customer?: Customer;
  lines: InvoiceLine[];
  totals: InvoiceTotals;
  invoiceType: "STANDARD" | "SIMPLIFIED";
}
```

---

# XML Generation Rules

The AI should:
- generate deterministic XML
- preserve namespace order
- preserve canonical formatting
- separate XML builder from signing logic

Never:
- mutate XML after signing
- insert dynamic whitespace after hashing

---

# UBL Strategy

Preferred approach:
1. create normalized invoice object
2. map domain object to UBL structure
3. serialize XML
4. canonicalize XML
5. hash XML
6. sign XML
7. append QR
8. submit to API

---

# Invoice Hashing

Use:
- SHA256

Implementation:
```ts
createHash("sha256")
```

The AI should understand:
- hashing must happen AFTER canonicalization
- even whitespace changes break signatures

---

# Signing Requirements

ZATCA requires:
- XAdES-BES signatures
- ECDSA secp256k1 certificates

The AI should:
- isolate signing logic
- keep signatures deterministic
- avoid frontend signing

# Queue-Based Processing

## Required Workflow

1. invoice created
2. enqueue invoice job
3. worker generates XML
4. worker signs invoice
5. worker submits invoice
6. worker stores response
7. worker retries failures

Reason:
ZATCA outages should not break checkout flows.

---

# Critical Concurrency Problem

## PIH Race Conditions

Multiple invoices generated simultaneously can corrupt:
- PIH chain
- ICV sequence

The AI should implement:
- transactional locking
- sequence tables
- mutexes
- queue serialization

Recommended:
PostgreSQL row locking.

---

# Recommended PostgreSQL Strategy

Use:

```sql
SELECT ... FOR UPDATE
```

when generating:
- ICV
- PIH

Never:
- compute these values in memory only

# Retry Strategy

# Error Handling
